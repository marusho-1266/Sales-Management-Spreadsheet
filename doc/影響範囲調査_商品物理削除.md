# 商品物理削除時の他機能への影響調査

**調査日**: 2026-03-06  
**対象**: 在庫管理シートにおける商品行の物理削除（`deleteRow()`）が他機能に与える影響

---

## 1. 調査概要

商品削除機能（物理削除方式）の実装にあたり、在庫管理シートから商品行を削除した場合に、関連する各機能へどのような影響があるかをコードベースから調査した。

---

## 2. 商品IDの参照元一覧


| モジュール      | ファイル                       | 参照方法                                   | 影響   |
| ---------- | -------------------------- | -------------------------------------- | ---- |
| 在庫管理       | InventoryManagement.gs     | 行番号・列インデックス                            | 削除対象 |
| 売上管理       | SalesManagement.gs         | 値として保存（商品ID, 商品名等）                     | 影響なし |
| 売上管理       | JoomDataTransform.gs       | 商品ID/SKUで検索・在庫更新                       | 注1   |
| 利益計算       | SheetInitializer.gs        | B2ドロップダウン・loadProductDataFromInventory | 注2   |
| 価格履歴       | PriceHistory.gs            | 商品IDで行検索・getProductNameById            | 注3   |
| Joom CSV出力 | JoomCsvExport.gs           | getDataRange()で全商品取得                   | 影響なし |
| スクレイピング更新  | WebScrapingDirectUpdate.gs | 仕入れ元URLでマッチ・行番号で更新                     | 影響なし |
| 新商品追加      | ProductInputForm.gs        | getNextProductId（最大ID+1）               | 影響なし |


---

## 3. 影響の詳細

### 3.1 在庫管理シート内


| 項目                         | 内容                                                                                                                                       |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **利益計算式（Z列）**              | 各行に `=I{row}-(H{row}+S{row}+V{row}+W{row}+X{row}+Y{row})` が設定されている。`deleteRow()` で行を削除すると、削除行より下の行が自動で繰り上がる。残存行の式は各セルに紐づいているため、**問題なし**。 |
| **onInventorySheetEdit**   | 編集トリガー。削除した行は編集されないため、**影響なし**。                                                                                                          |
| **setupProfitCalculation** | 初期化時に全データ行に式を設定。削除後の行構成に対して再実行する場合のみ関係し、通常運用では**影響なし**。                                                                                  |


### 3.2 売上管理シート


| 項目           | 内容                                                                                      |
| ------------ | --------------------------------------------------------------------------------------- |
| **売上データ**    | 商品ID・商品名・SKU等は**値**として保存されており、数式参照ではない。在庫管理から商品を削除しても売上データは**そのまま残る**。過去の売上履歴保持の観点から妥当。 |
| **注文入力フォーム** | `getInventoryDataForSelection()` が在庫管理の「在庫あり」商品のみ取得。削除した商品は含まれなくなる。**意図通りの動作**。         |


### 3.3 Joom注文連携（JoomDataTransform.gs）


| 項目                                | 内容                                                                                                            |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **getProductInfoFromInventory**   | 商品ID/SKUで在庫管理を検索。削除済み商品は見つからず、`productName: '商品名不明（在庫管理シートに未登録）'` として売上に登録される。**想定内**。                        |
| **updateInventoryFromJoomOrders** | 商品ID/SKUでマッチして在庫減算。削除済み商品はマッチしないため更新されない。**注意**: 削除後にその商品への新規Joom注文が来た場合、在庫は減算されず、売上には「商品名不明」で登録される。運用上の判断事項。 |


### 3.4 利益計算シート（SheetInitializer.gs）


| 項目                       | 内容                                                                                                                                                                     |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **B2 商品IDドロップダウン**       | `requireValueInRange(productIdRange)` で在庫管理の A2:A(lastRow-1) を参照。削除後は範囲が変わり、削除したIDは範囲外になる。**対策必要**: 削除直後に `refreshProfitSheetProductIdDropdown()` を呼び、ドロップダウンを再設定すること。 |
| **B2 に削除した商品IDが残っている場合** | データ検証は「新規入力時」のみチェックするため、既に入力済みの削除済みIDはそのまま残る。`loadProductDataFromInventory()` 実行時に「商品IDが見つかりません」となる。**対策**: 削除対象が利益計算で選択中だった場合、B2をクリアするか、別商品を促す。                       |


### 3.5 価格履歴シート（PriceHistory.gs）


| 項目                                | 内容                                                                                |
| --------------------------------- | --------------------------------------------------------------------------------- |
| **syncPriceHistoryFromInventory** | 在庫管理に存在する商品のみ処理。削除した商品はスキップされる。**問題なし**。                                          |
| **既存の価格履歴行**                      | 削除した商品の価格履歴行は**価格履歴シートに残る**。`syncPriceHistoryFromInventory` は追加・更新のみで削除処理は行わない。   |
| **getProductNameById**            | 在庫管理から商品名を取得。削除後は空文字を返す。価格履歴の既存行には過去の商品名が残るため、表示上は**問題なし**。                       |
| **方針**                            | 価格履歴は履歴として残す設計なら、削除した商品の行が残るのは許容範囲。削除する仕様にする場合は、商品削除処理内で価格履歴シートから該当行を削除する追加実装が必要。 |


### 3.6 Joom CSV出力（JoomCsvExport.gs）


| 項目                    | 内容                                                           |
| --------------------- | ------------------------------------------------------------ |
| **getTargetProducts** | `getDataRange().getValues()` で在庫管理を取得。削除した商品は含まれない。**問題なし**。 |
| **updateJoomStatus**  | 商品IDで在庫管理の行を検索して連携ステータスを更新。削除した商品はマッチしないため更新対象外。**問題なし**。    |


### 3.7 スクレイピング結果更新（WebScrapingDirectUpdate.gs）


| 項目                         | 内容                                                                                                        |
| -------------------------- | --------------------------------------------------------------------------------------------------------- |
| **updateInventoryFromCsv** | 仕入れ元URLで在庫管理とCSVをマッチ。マッチした行の行番号で更新。削除後は次回実行時に `getDataRange()` で最新状態を読み直すため、削除済み商品はマッチせずスキップされる。**問題なし**。 |


### 3.8 新商品追加（ProductInputForm.gs）


| 項目                                      | 内容                                                                                        |
| --------------------------------------- | ----------------------------------------------------------------------------------------- |
| **getNextProductId**                    | 在庫管理の商品ID列から最大値を取得し +1。削除で行数は減るが、最大IDは残存行から再計算される。全削除時は `maxNumber=0` となり次回IDは1。**問題なし**。 |
| **refreshProfitSheetProductIdDropdown** | 新規保存成功時に呼び出し。商品削除時も同様に呼ぶことで、利益計算のドロップダウンと整合する。                                            |


### 3.9 注文データ入力（SalesManagement.gs）


| 項目                               | 内容                                                |
| -------------------------------- | ------------------------------------------------- |
| **getInventoryDataForSelection** | 在庫管理の全データを読み、「在庫あり」の商品のみ返す。削除した商品は含まれない。**問題なし**。 |


---

## 4. 要対応事項まとめ


| 優先度    | 項目            | 対応内容                                             |
| ------ | ------------- | ------------------------------------------------ |
| **必須** | 利益計算B2ドロップダウン | 削除後に `refreshProfitSheetProductIdDropdown()` を呼ぶ |
| **推奨** | 利益計算B2の選択中商品  | 削除対象がB2で選択されていた場合、B2をクリアするか警告を出す                 |
| **任意** | 価格履歴          | 削除した商品の価格履歴行を残すか削除するかを仕様で決定                      |
| **任意** | Joom注文連携      | 削除後に同商品への新規注文が来た場合の扱いを運用で定義                      |


---

## 5. 物理削除の実行順序（推奨）

1. 商品IDの入力（プロンプトダイアログで指定）
2. 削除対象の検証（存在チェック）
3. 売上履歴の有無チェック（任意：警告表示）
4. 利益計算B2が削除対象商品IDかチェック → 該当時はB2をクリア
5. 在庫管理シートの該当行を `deleteRow()` で削除
6. `refreshProfitSheetProductIdDropdown()` を実行
7. （仕様により）価格履歴シートから該当商品IDの行を削除

---

## 6. 結論

物理削除を採用した場合、**以下の2点を削除処理に組み込めば他機能への悪影響は回避できる**。

1. **削除直後に `refreshProfitSheetProductIdDropdown()` を実行する**
2. **削除対象が利益計算B2で選択されていた場合はB2をクリアする**

その他の機能は、商品IDや在庫データをその時点のシート状態から都度取得しているため、物理削除後も整合した動作となる。
